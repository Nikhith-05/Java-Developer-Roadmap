Salary negotiation script

  • Always give a range, not a single number. "My expectation is 12–15 LPA based on market data."
  • Research: Glassdoor, AmbitionBox, LinkedIn Salary for the exact company+role+city.
  • Never reveal your current/previous CTC first. "Let me understand the full scope of the role first."

Resume strategy for freshers targeting 10+ LPA

Most fresher resumes are rejected in under 10 seconds because they list skills without context, use dense templates, and include irrelevant content. A clean, ATS-friendly 1-page resume with 2–3 strong projects beats a 3-page resume with 10 weak ones.

Resume structure (in this order)

Section Content Length
Header Name, phone, email, LinkedIn, GitHub (no photo, no address) 3 lines
Summary (optional) 1–2 lines: "Java backend developer with strong DSA skills and 2 production-grade projects" 2 lines max
Skills Java, Spring Boot, SQL, Git, DSA, REST APIs, Docker (only what you know) 2–3 lines
Projects (most important) 2–3 projects with bullet points: what you built, tech used, scale/impact 1/3 of resume
Education Degree, college, CGPA (include only if 7.0+), year 3–4 lines
Achievements LeetCode rating, hackathon wins, certifications, open source contributions 3–5 bullets
Certifications Only add if relevant: AWS Cloud Practitioner, Oracle Java Cert, Spring cert 1–2 lines

How to write project bullets (the difference-maker)

Bad: "Built an e-commerce website using Spring Boot and MySQL"

Good: "Developed a RESTful e-commerce backend with Spring Boot, handling 10+ API endpoints, JWT authentication, and Redis caching reducing DB load by 40% in local load tests. Deployed on Railway with CI/CD via GitHub Actions."

  • → Always include: what problem it solves, tech stack, a metric or scale indicator
  • → GitHub link must be live with a README that explains setup and architecture
  • → Use action verbs: Designed, Built, Implemented, Optimised, Deployed

ATS tips

  • Use a clean single-column or two-column (left: main content) template. No tables, no text boxes.
  • Include keywords from the job description: "Spring Boot", "REST API", "microservices", "Java 17".
  • Use standard section headings: "Projects", "Education", "Skills" — not creative names.
  • Submit as PDF unless docx explicitly asked for.
  • Keep to exactly 1 page. Experienced can do 2; fresher must do 1.

Common mistakes that kill 10+ LPA chances

Tutorial hell: Watching 100 hours of tutorials without building anything. Build after every concept, even if it's small.

Skipping DSA: "I'll learn Spring Boot first, DSA later." Wrong. Service companies hire without DSA; product companies don't.

Copy-paste projects: Following YouTube tutorials step-by-step and putting the result on resume. Recruiters have seen 1000 identical todo apps.

No deployment: Projects that only run on localhost aren't impressive. Deploy to Railway, Render, or AWS free tier.

Weak GitHub: Private repos, no README, no commit history, no description. Your GitHub is your portfolio — treat it like one.

Only applying online: 80% of hirings involve referrals. Spend at least 30% of job-search time on networking.

Neglecting communication: In interviews, thinking silently and then presenting the solution is a red flag. Talk through your approach always.

The brutal truth about 10+ LPA as a fresher

Is 10+ LPA realistic?

Yes — but for the top 5–10% of freshers, not average ones. In India 2026, companies like Razorpay, PhonePe, Zepto, Swiggy, Amazon, and Atlassian regularly hire freshers at 12–35 LPA. These are competitive — thousands apply, 10–30 get hired. But if you follow this roadmap with full discipline, you are genuinely competitive for these roles.

What separates selected vs rejected candidates?

It comes down to three things. First, DSA depth — most rejected candidates solve 80 easy problems and stop. Hired candidates solve 250–350 problems including mediums and understand patterns, not solutions. Second, project quality — hired candidates can talk about every design decision in their project for 30 minutes. Rejected candidates built it from a tutorial. Third, communication — in every round, the ability to think out loud and articulate trade-offs is as important as the correct answer.

How much effort is actually required?

6 hours of focused work daily, 7 days a week, for 10–12 months. Not 6 hours of watching tutorials — 6 hours of coding, building, and solving. That is roughly 2000+ hours of quality practice. Most people who "try for 6 months" actually put in 800–900 hours because of breaks, half-attention, and comfort-zone problems only. The ones who make it are simply the ones who don't stop.

Your biggest risk

Not lack of talent — lack of consistency. 90% of people who start this journey quit in month 2–3 when the material gets hard (DSA trees, backtracking). If you get through month 4, you are already in the top 20% of candidates. If you get through month 7, you are in the top 5%.
